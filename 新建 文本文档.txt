 
本科毕业设计（论文）开题报告
 

题目：	融合RAG与多模态大模型
的智能健康助手设计与实现


 
1 选题的目的和意义
1.1 选题的背景与目的
1.1.1	选题背景
近年来，随着公众健康意识的显著提升，移动健康管理应用已成为大众日常自我监测与健康管理的重要工具。然而，当前市场上的主流健康类App普遍存在一定的局限性：一方面，大多数应用停留在单一数据的被动记录层面（如仅利用传感器统计步数或通过手动输入计算卡路里），缺乏对用户饮食、运动及多项生理指标等多源数据的综合处理与深度挖掘能力；另一方面，传统App提供的健康干预建议往往依赖预设的静态规则，呈现出千篇一律的特征，难以满足不同用户个性化、深层次的健康指导需求。
与此同时，生成式人工智能（AIGC）技术，特别是多模态大语言模型（Multimodal LLM）的飞速发展，为跨越上述技术瓶颈提供了新契机。然而，直接将通用大模型应用于医疗健康领域往往面临“AI幻觉”的风险，缺乏垂直领域的专业性和严谨性。
1.1.2	选题目的
基于上述背景，本课题旨在设计并实现一款基于多模态大模型与RAG（检索增强生成）技术的Android端个人健康管理系统。本课题的开发目的主要包括：
1.	打破传统“只记录不诊断”的局限：利用多模态大模型的视觉理解与跨模态融合分析能力，实现对用户饮食图片的智能识别及多源健康数据的融合分析。
2.	解决健康AI问答的“幻觉”痛点：通过引入RAG技术，构建本地化的专业健康向量知识库，为大模型提供精准的外部知识源，确保健康咨询与问答的专业性、准确性与可靠性。
3.	实现个性化健康干预体系：基于用户的当日运动量、饮食摄入等多维数据，通过科学的Prompt工程设计，自动生成定制化的个人健康日报和干预方案。
1.2 选题的技术现状
随着智能手机传感器的普及与Android生态的成熟，移动健康管理（mHealth）应用已进入快速发展期。传统的健康管理App主要侧重于数据的采集与可视化，例如张宪文等（2020）基于Android系统设计的运动健康App，利用手机自带传感器实现了计步及卡路里消耗的自动记录，验证了移动端基础生理参数监测的便捷性[4]。然而，当前主流应用普遍存在“重记录、轻指导”的局限。一方面，单纯的数据记录已难以满足年轻一代对健康管理深度与趣味性的需求，正如潘雨忱和袁徐庆（2025）指出，当前市场缺乏深层次的智能交互体验[5]；另一方面，肖欣雨等（2024）的研究也表明，现有应用在用户体验上存在断层，缺乏对多源数据（如饮食习惯）的智能化分析，导致系统提供的健康建议往往千篇一律，缺乏针对性[1]。

近年来，以大语言模型为代表的生成式人工智能技术展现出强大的自然语言理解与推理能力，为突破传统健康App的“指导瓶颈”提供了新路径。但在垂直的医疗健康领域，通用大模型容易产生“幻觉”现象，输出看似合理却缺乏科学依据的错误信息。为此，学术界开始探索将外部知识库与大模型结合的技术路线。赵海燕等（2025）提出了一种融合知识图谱和大模型的医疗智能问答方法，通过引入外部结构化知识，有效提升了模型回答的准确率与可解释性[2]；邢雪凯（2025）构建的医疗问答助手同样验证了在特定医疗场景下，利用检索增强生成（RAG）等技术约束大模型，是降低幻觉、提升专业性的有效途径[3]。

综上所述，虽然移动健康应用在数据采集上已较为成熟，且大模型在医疗问答领域的技术探索也取得了初步成果，但将多模态大模型与RAG技术轻量化落地至Android移动端，并结合视觉理解实现饮食与运动综合管理的工程实践仍相对匮乏，这也是本课题切入的核心技术背景。
1.2.1	选题的意义
本课题的研究旨在解决当前健康管理应用中“智能化不足”与“大模型垂直落地难”的痛点，兼具重要的理论探索价值与实际应用意义。

在理论与技术层面，本课题探索了在Android移动端中高效调度云端多模态大模型与本地RAG检索技术的协同工作机制。这不仅为大模型在轻量级移动端的工程落地提供了可参考的架构范式，同时也验证了将视觉理解（如饮食图片识别）与文本推理（如健康建议生成）相融合的多模态技术，在提升个人健康管理精度与多维数据融合方面的可行性。

在实际应用层面，本系统的开发将极大提升普通用户的健康管理体验。通过引入多模态AI视觉识别，系统能够代替繁琐的手动录入，有效降低用户记录饮食的门槛；同时，基于RAG技术构建的本地健康知识库，确保了系统问答与干预建议的专业性与安全性，使用户能够低成本获得类似“私人医生”的实时指导。此外，区别于传统App机械式的图表反馈，本系统能够根据用户当日的综合运动与饮食数据，自动生成定制化的健康日报，从而更科学、精准地引导用户改善生活习惯，对促进大众健康具有积极的现实意义与应用示范价值。
2 题目的主要内容 
本部分将围绕基于多模态大模型与RAG技术的Android端个人健康管理系统，详细阐述本课题的具体研究内容与开发规划。下文将从整体工作任务的切分入手，逐步深入到系统的总体架构设计，并对各个核心功能模块的业务逻辑进行拆解说明，最后明确本毕业设计需达成的预期结果与最终交付物，从而为后续的系统开发与论文撰写提供清晰的执行路径。
2.1 任务概述
本课题旨在设计并开发一款基于多模态大模型与RAG（检索增强生成）技术的Android端个人健康管理系统。课题的主要任务包括四个方面：首先是需求调研与技术选型，分析现有健康类应用在数据记录与健康咨询方面的局限，确定基于Android Jetpack组件库、Retrofit网络框架及主流大模型API的技术架构；其次是系统架构与数据库设计，规划基于MVVM模式的客户端与RESTful API风格的后端服务，并重点设计多模态数据分析与知识库问答的独立数据流向；第三是核心功能模块的开发，涵盖Android端传感器数据采集（步数、站立）、多模态饮食图片识别、基于本地向量知识库的RAG智能问答，以及基于提示词工程（Prompt Engineering）的个性化健康报告生成与图表可视化；最后是系统测试与优化，重点评估AI接口的响应速度、多模态识别的准确率及RAG生成内容的可靠性，并据此优化提示词策略，最终完成毕业设计论文的撰写与成果交付。
2.2 设计的总体结构描述
本系统采用典型的“客户端-服务端-AI大模型”三层架构进行设计，通过合理的职责划分确保系统的可扩展性与高维护性。
在客户端（Android），系统采用MVVM架构模式。View层负责UI界面的展示与用户交互（如拍照、图表渲染）；ViewModel层负责处理界面的业务逻辑并管理UI状态；Model层结合Retrofit框架负责与后端的网络通信以及本地传感器（如计步器）的数据获取。
在服务端（后台服务层），系统采用RESTful API风格设计接口，主要承担请求路由、用户鉴权、数据持久化以及大模型请求代理的核心逻辑。服务端不仅连接传统的结构化数据库（用于存储用户基础信息与历史运动/饮食数据），还接入了向量数据库（用于构建本地健康知识库）。
在AI层，系统设计了双重数据处理流向：一条流向多模态大模型，用于处理客户端上传的饮食图片，进行视觉识别与热量估算；另一条流向RAG（检索增强生成）模块，当用户发起健康咨询时，服务端先从向量知识库中检索出相关的专业医疗健康知识，再连同用户的提问一起构建Prompt发送给大语言模型，从而生成严谨、低幻觉的专业回答。
2.2.1	设计的各个功能模块描述
根据系统总体架构，本系统主要划分为以下五个核心功能模块：
1.	数据采集与记录模块：负责多维度的用户健康数据追踪。一方面通过调用Android系统底层的传感器API，实现全天候的步数统计与站立状况记录；另一方面提供便捷的饮食记录入口，允许用户通过手机摄像头拍照或相册上传饮食图片。
2.	多模态饮食识别模块：接收用户上传的饮食图片，调用多模态大模型的视觉理解接口，智能识别图片中的食物种类与大致分量，并结合后台数据库自动估算摄入的热量与宏量营养素，免去用户手动查询和录入的繁琐过程。
3.	智能健康问答模块：作为系统的核心交互模块，提供全天候的AI健康咨询服务。该模块利用预先构建的本地向量健康知识库，采用检索增强生成技术，拦截用户的自然语言提问，检索高相关性的医学片段作为上下文，确保回答的科学性与专业性。
4.	健康报告生成与可视化模块：汇总用户的当日运动消耗（步数、基础代谢等）与饮食摄入数据，计算热量收支缺口。通过图表组件直观地呈现数据的动态变化趋势。同时，结合当日综合数据构建动态Prompt，调用大模型生成逻辑通顺、建议合理的个性化健康日报与干预方案。
5.	后台服务与数据管理模块：提供稳定高效的后端API支持，涵盖用户注册登录鉴权、历史健康数据的增删改查、以及向量数据库的维护与更新，保障多端数据的一致性与AI接口调用的安全性。
2.3 设计的预期结果
本课题通过系统化的设计与开发，预期达成以下成果：

开发出一款能够流畅运行于主流Android设备的移动应用。该应用需准确调用传感器记录步数，并精准识别用户上传的饮食图片；能够根据用户的当日综合健康数据，生成逻辑通顺、科学合理的个性化健康日报；通过直观的图表组件清晰展示用户的健康趋势与热量收支。

3 设计的技术路线
3.1 开发工具和开发环境
为了保证系统的高效开发与稳定运行，本项目将采用以下开发工具与运行环境：
开发设备：Windows11 个人计算机
测试设备：Redmi K60 Android智能手机

客户端开发：使用Android Studio作为集成开发环境，Kotlin为主要开发语言，结合Android SDK、Jetpack 组件库以及 Retrofit 网络请求框架进行开发。
服务端与AI开发：考虑到大模型生态与RAG技术的适配性，后端拟采用 Python 语言结合 FastAPI 或 Flask/Django 轻量级框架进行开发，以方便对接大语言模型API（如智谱GLM、通义千问或OpenAI等）及开源大模型框架。
数据库系统：采用关系型数据库（如 MySQL 或 PostgreSQL）存储用户基本信息与常规生理数据；采用向量数据库（如 ChromaDB、FAISS 或 Milvus）存储本地医疗健康知识库的向量化数据。
其他辅助工具：Git（版本控制）、Postman（接口测试）、Markdown（撰写文档）等。
3.1.1	设计的思路与方法
系统整体采用“自顶向下”的设计思路，遵循“高内聚、低耦合”的软件工程原则。客户端严格按照 MVVM 架构进行分层设计，实现UI与业务逻辑的解耦；服务端采用前后端分离的 RESTful API 设计。在核心AI交互方面，采用“双流向”处理机制：对于用户的饮食记录，调用多模态大模型的视觉接口提取热量与营养信息；对于健康问答与报告生成，先通过 RAG 模块检索本地医学知识片段，再结合用户的运动/饮食结构化数据构建动态 Prompt，最后交由大模型生成结果并返回给移动端。

主要使用文献研究以及原型设计与迭代开发的方法。查阅国内外关于 mHealth、大语言模型应用及 RAG 技术的相关文献，掌握前沿技术动态，为系统设计提供理论支撑。前期使用原型工具设计应用交互界面；在编码阶段采用敏捷迭代的方法，先跑通“客户端-服务端-大模型”的基础链路，再逐步完善多模态识别与RAG检索细节。
3.2 可行性分析
技术可行性方面，当前 Android 开发生态十分成熟，Jetpack 组件和 Retrofit 框架能够高效构建稳定的客户端应用；移动端调用手机传感器（如计步）已有标准的 API 支持。同时，各大AI厂商均提供了完善的多模态大模型 API 接口，且 LangChain 等开源框架极大地降低了 RAG 系统的搭建门槛。作为软件工程专业的本科生，具备上述技术栈的学习与整合能力，技术路线上完全可行。
经济可行性方面，本项目开发所需的软件工具（如 Android Studio、Python、MySQL、向量数据库等）均为开源或免费软件。大语言模型的 API 调用在前期开发与测试阶段可通过开发者免费额度或低成本的 Token 计费方式解决，无需采购昂贵的硬件服务器或 GPU 算力，开发成本极低，经济上完全可行。
操作与用户可行性方面，系统主要面向普通大众，Android 客户端界面设计遵循移动端交互规范，操作简便。用户只需通过手机拍照和日常携带手机即可完成复杂的数据记录，极大地降低了用户的使用门槛。因此，系统具有很高的实用性与操作可行性。
3.3 重点与难点分析
研究重点主要包括三个方面。其一是多模态数据的融合与处理。具体为如何有效打通 Android 端硬件（摄像头采集的非结构化图像数据、传感器采集的结构化运动数据）与云端大模型之间的传输链路，并让模型准确识别食物种类及预估热量。
其二为构建基于RAG的健康知识库。如何收集可靠的医疗健康文本数据，对其进行合理的文本切分与向量化，并存入向量数据库以供高效检索。
其三为Android端开发中的MVVM架构深度应用。保证在频繁进行网络请求与大量数据可视化（如图表渲染）时，UI 主线程的流畅性与应用状态的稳定性，从而让用户有良好的使用体验。

研究难点主要包括两个方面。其一为Prompt Engineering（提示词工程）的设计与优化。大语言模型的输出具有一定的不确定性。如何将用户的历史运动数据、当日摄入热量以及 RAG 检索到的医学知识，巧妙地拼接成高质量的 Prompt，使模型能够稳定输出逻辑严密、格式规范（如 JSON 格式以便 Android 端解析）的个性化健康报告，是本课题的一大难点，也是本课题的核心着力点。
其二为响应延迟的平衡。在引入 RAG 技术提升专业性的同时，多轮检索和长文本生成会导致接口响应时间变长。如何在移动端通过异步处理、加载动画或流式输出来优化用户体验，同时确保医疗健康建议的严谨性与低幻觉，是系统工程实现中的关键挑战。
